# Step3 Fixed-Topology Multi-Seed DFL Experiment

> **Status:** Experimental design specification  
> **Recommended experiment directory:**  
> `surrogate_experiment_results/Step3/`

## Advisor-facing summary

This experiment treats the KEP topology as an outer experimental block. For each selected topology, the vertex set, arc set, node types, and feasible cycle/chain structure are fixed. Multiple training datasets are generated by resampling complete \((X,y)\) instances under that same topology. The training size varies over \(n\in\{50,100,500\}\), while the training seed varies over 1,000 repetitions. All methods and all training sizes associated with a topology are evaluated on the same fixed validation and test sets generated under that topology. The experiment first screens a broad topology bank using a reduced seed budget, then confirms a smaller selected-and-matched topology set with the full multi-seed protocol.

## 1. Background and motivation

The existing Step2 resampling experiment varies `subset_seed` inside a fixed pool of many KEP graphs. Therefore, changing the seed changes which graph topologies enter the training set. That experiment measures robustness to **training-graph subset composition**.

The present experiment answers a different and more controlled question:

> When the KEP feasible set is fixed, how much do training-sample randomness and training-set size affect 2stage and SPO+, and on which fixed graph instances is the DFL advantage stable across seeds?

The main separation is:

```text
Existing subset-resampling experiment:
    seed -> different collection of graph topologies

This fixed-topology experiment:
    topology is fixed
    seed -> different (X, y) sample realizations on that topology
```

This experiment is not intended to claim that a raw topology alone causes DFL to outperform 2stage. The paper-safe interpretation is:

```text
DFL performance is graph-instance / feasible-solution-landscape conditioned.
A fixed-topology multi-seed protocol measures whether an observed advantage is
stable to sample generation rather than being an isolated seed effect.
```

## 2. Experimental objectives

### 2.1 Primary objective

For a fixed KEP topology \(G_k\), quantify the distribution of the decision-quality difference between 2stage and SPO+ over repeated training datasets:

\[
\Delta_{k,n,s}
=
\operatorname{Gap}_{\text{2stage},k,n,s}
-
\operatorname{Gap}_{\text{SPO+},k,n,s},
\]

where:

- \(k\) indexes topology;
- \(n\in\{50,100,500\}\) is training size;
- \(s\in\{1,\ldots,1000\}\) is training seed;
- positive \(\Delta_{k,n,s}\) means SPO+ has lower decision gap than 2stage.

The primary question is:

> Under the same topology, fixed evaluation set, and controlled implementation seeds, is the SPO+ improvement over 2stage stable across independently generated training datasets?

### 2.2 Secondary objectives

1. Measure how the 2stage–SPO+ difference changes with training size.
2. Identify fixed graph instances that are robustly:
   - helpful for SPO+;
   - neutral;
   - harmful for SPO+.
3. Relate those outcomes to:
   - topology descriptors;
   - feasible-set geometry;
   - predicted candidate-list ambiguity;
   - rank reversal and decision-critical edges.
4. Select a small number of statistically confirmed graph examples for mechanism dissection.

### 2.3 Non-objectives

This experiment does **not** directly establish:

- causal effects of graph density, cycle count, or another topology statistic;
- generalization from one topology to every unseen topology;
- clinical realism of the synthetic \((X,y)\) generator;
- that a graph selected after screening represents the average KEP graph.

---

## 3. Formal unit of observation

### 3.1 Fixed topology

For topology \(G_k\), keep fixed:

```text
vertex identities and vertex types
directed compatibility arcs
non-directed donor / paired-donor structure
maximum cycle length
maximum chain length
arc ordering
feasible cycle and chain candidates
solver constraints
```

Recommended solver limits:

```text
max_cycle = 3
max_chain = 4
```

Let the fixed feasible decision set be:

\[
\mathcal Z(G_k).
\]

No training seed may add or remove a vertex or arc. The topology hash and feasible-set hash must remain identical across all samples associated with \(G_k\).

### 3.2 One \((X,y)\) pair

One sample is a **complete optimization instance on the fixed topology**, not a single edge.

For topology \(G_k\) with \(m_k\) arcs and \(p\) features:

\[
X_{k,i}\in\mathbb R^{m_k\times p},
\qquad
y_{k,i}\in\mathbb R^{m_k}.
\]

Here:

- \(X_{k,i}\) is the full edge-feature matrix for sample \(i\);
- \(y_{k,i}\) is the full true edge-reward vector for sample \(i\);
- every row uses the same canonical arc ordering.

The current two-feature probe is:

\[
\widehat y_{k,i,e}
=
\theta_1 u_{k,i,e}
+
\theta_2 c_{k,i,e},
\]

where \(u\) is the utility-style feature and \(c\) is the recipient-cPRA-style feature.

The predicted decision is:

\[
\widehat z_{k,i}(\theta)
\in
\arg\max_{z\in\mathcal Z(G_k)}
\widehat y_{k,i}^{\top}z.
\]

The oracle decision is:

\[
z^*_{k,i}
\in
\arg\max_{z\in\mathcal Z(G_k)}
y_{k,i}^{\top}z.
\]

### 3.3 Critical interpretation of “fixed topology”

Fixed topology means:

```text
fixed vertices + fixed arcs + fixed feasible set
```

It does **not** require every \(X\) and \(y\) realization to be identical.

The formal advisor protocol requires different training seeds to produce distinct \((X,y)\) samples. Therefore:

- Step2b can vary across seeds only if \(X\) is resampled;
- Step2c can vary through both \(X\)-resampling and multiplicative label noise;
- keeping \(X\) fixed and changing only Step2c label noise is a useful ablation, but it is not the full \((X,y)\)-resampling protocol;
- keeping \(X\) fixed in noiseless Step2b would produce identical samples and is invalid as a multi-seed experiment.

## 4. Data-generating regimes

### 4.1 Primary regime: Step2c high misspecification

Recommended formal regime:

```text
regime = step2c_poly_d8_mult_eps050
degree = 8
kappa = 3
epsilon_bar = 0.5
```

For a sampled feature matrix \(X\), define for each arc \(e\):

\[
b_e=10u_e+5c_e,
\]

\[
\mu_G=\frac{1}{|E_G|}\sum_e b_e,
\qquad
q_e=\frac{b_e}{\mu_G+\delta},
\]

\[
r_e^{(d)}=(q_e+\kappa)^d-\kappa^d,
\]

\[
m_e^{(d)}
=
\mu_G
\frac{r_e^{(d)}}
{\frac{1}{|E_G|}\sum_{e'}r_{e'}^{(d)}+\delta}.
\]

Then draw:

\[
\eta_e\sim\operatorname{Uniform}(1-\bar\epsilon,1+\bar\epsilon)
\]

and set:

\[
y_e=\max(0,m_e^{(d)}\eta_e).
\]

This is the main regime because it contains:

```text
nonlinear model misspecification
+
sample-level stochastic uncertainty
```

### 4.2 Secondary regime: Step2b noiseless polynomial

Recommended secondary regime:

```text
regime = step2b_poly_d8
degree = 8
kappa = 3
```

The label is:

\[
y_e=\max(0,m_e^{(d)}).
\]

Because this label is deterministic conditional on \(X\), the context generator must produce distinct \(X\) realizations across sample seeds.

### 4.3 Optional moderate-misspecification checks

After the high-degree experiment is stable:

```text
step2b_poly_d4
step2c_poly_d4_mult_eps050
```

These should be secondary checks, not the first computational target.

### 4.4 Required context-generator interface

The formal experiment must implement and lock a context generator:

```python
X = sample_context(
    topology_template=G_k,
    rng=context_rng,
    generator_config=context_config,
)
```

The exact generator must be documented in a versioned configuration file. It must satisfy:

1. fixed matrix shape for a fixed topology;
2. fixed arc ordering;
3. recipient-level variables remain recipient-consistent;
4. all feature ranges are valid;
5. repeated calls with the same seed are bitwise reproducible;
6. different sample seeds produce non-identical \(X\) with high probability;
7. topology and feasible set do not change.

Possible implementations include:

```text
A. Empirical bootstrap generator
   Sample node-level and edge-level covariates from an empirical source bank,
   respecting node/edge roles and recipient-level consistency.

B. Base-feature perturbation generator
   Perturb the original topology-template covariates with a documented,
   bounded stochastic model while preserving valid ranges.

C. Existing synthetic KEP attribute generator with adjacency frozen
   Reuse the project generator for attributes, but prevent any change to
   vertices, arcs, and feasible structures.
```

The formal run must not mix generators. The chosen generator and all parameters must be frozen after the pilot.

### 4.5 Label-only Step2c ablation

An optional ablation may keep \(X\) fixed and vary only the Step2c noise:

```text
fixed X
fixed topology
different label_seed -> different y
```

This ablation isolates label noise but must be reported separately from the full \((X,y)\)-resampling experiment.

## 5. Randomness and controlled seeds

The experiment separates randomness into explicit namespaces.

| Source | Formal setting | Purpose |
|---|---:|---|
| `topology_id` | outer experimental unit | changes fixed graph instance |
| `train_seed` | 1 to 1000 | changes the generated training sample bank |
| `sample_index` | 0 to 499 | identifies a sample inside one nested bank |
| `context_seed` | derived deterministically | generates \(X\) |
| `label_seed` | derived deterministically | generates Step2c noise |
| `validation_seed` | fixed within `(topology, regime)` | creates fixed validation set |
| `test_seed` | fixed within `(topology, regime)` | creates fixed test set |
| `theta_seed` | fixed, default 42 | controls model initialization |
| `gurobi_seed` | fixed, default 42 | controls solver reproducibility |
| `bootstrap_seed` | fixed, default 42 | controls confidence-interval resampling |

### 5.1 Stable seed derivation

Do not use Python’s built-in `hash()`. Use SHA-256 or another stable hash:

```text
context_seed =
  stable_hash(
    "context",
    experiment_version,
    regime,
    topology_id,
    split_namespace,
    train_seed,
    sample_index
  )

label_seed =
  stable_hash(
    "label",
    experiment_version,
    regime,
    topology_id,
    split_namespace,
    train_seed,
    sample_index
  )
```

Recommended split namespaces:

```text
screen_train
screen_validation
screen_test
confirm_train
confirm_validation
confirm_test
```

This prevents accidental overlap even when integer seed values are reused.

## 6. Core experimental protocol

The formal confirmation protocol is:

```text
Choose K fixed topologies:
    G_1, G_2, ..., G_K

For each regime R:
    For each topology G_k:

        Freeze and audit the topology template:
            fixed vertices
            fixed arcs
            fixed feasible set
            fixed arc ordering
            fixed topology hash

        Build one fixed validation set:
            same topology G_k
            fixed validation seed namespace
            fixed validation size

        Build one fixed test set:
            same topology G_k
            fixed test seed namespace
            fixed test size
            shared by all methods, train seeds, and train sizes

        For train_seed s in {1, ..., 1000}:

            Generate one maximum-size training bank:
                D_train(k, s) = 500 (X, y) pairs

            Construct nested training sets:
                n = 50  -> first 50 pairs
                n = 100 -> first 100 pairs
                n = 500 -> all 500 pairs

            For train_size n in {50, 100, 500}:

                Train 2stage MSE:
                    same training pairs
                    fixed theta initialization
                    fixed hyperparameters

                Train SPO+:
                    same training pairs
                    same theta initialization
                    fixed solver settings
                    fixed hyperparameters

                Select checkpoints using only the fixed validation set.

                Evaluate both methods on the same fixed test set.

                Save:
                    model parameters
                    selected epoch
                    validation metrics
                    test metrics
                    per-test-pair decisions
                    configuration and data hashes
```

### 6.1 Nested training sets are mandatory

For a fixed `(topology, regime, train_seed)`, generate 500 samples once:

```text
D_50  = D_500[0:50]
D_100 = D_500[0:100]
D_500 = D_500[0:500]
```

This creates a paired sample-size comparison. Do not independently regenerate the 50-, 100-, and 500-sample datasets.

### 6.2 Fixed evaluation sets are mandatory

For a fixed `(topology, regime)`:

```text
validation set is identical across:
    methods
    train seeds
    train sizes

test set is identical across:
    methods
    train seeds
    train sizes
```

The fixed test set must never be used for:

```text
checkpoint selection
hyperparameter tuning
topology screening
choosing the context generator
choosing practical-effect thresholds
```

## 7. Recommended experimental phases

A direct run with large \(K\) and 1,000 seeds is expensive. Use a broad-screening / narrow-confirmation design.

## 7.1 Phase 0: implementation smoke test

Purpose:

```text
validate data schema
validate seed determinism
validate nested train sets
validate fixed topology
validate end-to-end training and evaluation
```

Recommended scale:

```text
K = 2 or 3
regime = Step2c d8 eps050
train_sizes = {50, 100}
train_seeds = 1..5
validation_size = 100
test_size = 200
epochs = short smoke setting
```

No scientific conclusion should be drawn from Phase 0.

## 7.2 Phase 1: broad topology screening

Purpose:

```text
find candidate helpful, neutral, and harmful graph instances
estimate computational cost
avoid committing 1,000 seeds to uninformative topologies
```

Recommended scale:

```text
K_screen = 100 to 400 topologies
primary regime = Step2c d8 eps050
train_size = 50
screen_train_seeds = 1..20 or 1..50
screen_validation_size = 500
screen_test_size = 1000
```

If compute is limited:

```text
K_screen = 100
screen_train_seeds = 25
```

If existing heldout400 graph descriptors are available, all 400 may be screened.

Screening output:

```text
candidate helpful graphs
candidate neutral graphs
candidate harmful graphs
matched controls
topology and feasible-set descriptors
compute-time statistics
```

Screening uses `screen_*` seed namespaces. It must not reuse the formal confirmation test set.

## 7.3 Phase 2: fixed-topology confirmation

Recommended formal setting:

```text
K_confirm = 10
train_sizes = {50, 100, 500}
confirm_train_seeds = 1..1000
validation_size = 1000
test_size = 5000
methods = {2stage, SPO+}
theta_seed = 42
gurobi_seed = 42
```

Suggested topology composition:

```text
3 to 4 screening-helpful candidates
2 to 3 neutral controls
2 to 3 harmful controls
topology-matched controls where possible
```

The final set must be locked in:

```text
configs/confirmation_topologies.json
```

before confirmation training begins.

### Training-count calculation

For \(K\) topologies:

\[
\text{training runs}
=
K\times 3\times 1000\times 2
=
6000K.
\]

Examples:

```text
K = 5  -> 30,000 training runs
K = 10 -> 60,000 training runs
K = 20 -> 120,000 training runs
```

Recommended choices:

```text
full 1,000-seed confirmation:
    K = 5 to 10

larger topology coverage with reduced seeds:
    K = 20, seeds = 200

broad discovery:
    K = 100 to 400, seeds = 20 to 50, train_size = 50
```

The default paper-facing plan is:

```text
screen broadly, confirm narrowly
```

## 7.4 Phase 3: mechanism dissection

Select 3–5 confirmed examples:

```text
one robust helpful case
one clean top-K promotion case
one deep reranking case
one harmful control
one neutral or both-poor control
```

For those graphs, run:

```text
predicted top-M candidate enumeration
true oracle top-M landscape
rank-reversal analysis
critical-edge signed contributions
candidate overlap / Jaccard analysis
feasible-set ambiguity diagnostics
```

Reuse shared logic from the existing decision-analysis and Step2c mechanism-audit utilities where possible.

## 8. Topology selection protocol

### 8.1 Topology source bank

The existing multi-graph datasets should be treated as:

```text
topology source bank
and/or
empirical feature-distribution source
```

They should not be used directly as the new experiment’s train/test split.

Candidate sources include:

```text
the existing 400 heldout graph population
the Step2 main2000 graph pool
a dedicated topology bank created from raw KEP instances
```

For every selected topology, export a self-contained immutable template.

### 8.2 Recommended screening strata

Avoid screening only the easiest or densest graphs. Cover variation in:

```text
number of vertices
number of arcs
density
reciprocity
2-cycle count
3-cycle count
number of feasible chains
number of feasible exchanges
conflict-graph density
largest conflict component
source/sink structure
```

### 8.3 Confirmation selection criteria

A graph is a candidate robust-helpful case when, on screening seeds:

```text
median Delta > 0
fraction(Delta > 0) is high
effect is not driven by one or two outlier seeds
```

A stronger pre-registered criterion may be:

```text
95% CI for mean Delta is above 0
and
fraction(Delta > 0) >= 0.80
```

A harmful control may satisfy:

```text
95% CI for mean Delta is below 0
or
fraction(Delta > 0) <= 0.20
```

A neutral control may satisfy:

```text
absolute median Delta is below a locked practical threshold
and
the confidence interval overlaps 0
```

The practical threshold must be chosen during the pilot and frozen before confirmation.

### 8.4 Avoiding cherry-picking

Do not select topologies using the final confirmation test set.

The valid sequence is:

```text
screening data
    -> select and lock topology set
    -> generate new confirmation validation/test sets
    -> run confirmation
    -> report every locked topology
```

Selected case-study graphs must be labeled as case studies, not population-average evidence.

## 9. Model training

### 9.1 Shared model class

Use the same linear probe for both methods:

\[
\widehat y_e = \theta_1 u_e+\theta_2 c_e.
\]

Do not change the hypothesis class between 2stage and SPO+.

### 9.2 2stage MSE

Training objective:

\[
\min_{\theta}
\frac{1}{n}
\sum_{i=1}^{n}
\frac{1}{m_k}
\left\|
X_{k,i}\theta-y_{k,i}
\right\|_2^2.
\]

Primary checkpoint rule:

```text
minimum fixed-validation MSE
```

### 9.3 SPO+

Train with the existing project SPO+ formulation on the same \((X,y)\) training pairs.

Primary checkpoint rule:

```text
minimum fixed-validation SPO+ loss
```

Optional diagnostic checkpoint:

```text
minimum fixed-validation decision gap
```

The diagnostic checkpoint must not replace the primary checkpoint in the main comparison.

### 9.4 Controlled optimizer settings

Recommended defaults:

```text
theta_seed = 42
gurobi_seed = 42
2stage optimizer = Adam
SPO+ optimizer = Adam
2stage epochs = 500
SPO+ epochs = 500
metric_stride = 10
```

Learning rates and any stabilization parameters must be selected in Phase 0/1 and frozen before Phase 2.

### 9.5 Pairing requirements

For every `(topology, regime, train_seed, train_size)`:

```text
2stage and SPO+ use exactly the same training pairs
2stage and SPO+ use exactly the same initial theta
2stage and SPO+ use the same fixed validation set
2stage and SPO+ use the same fixed test set
```

## 10. Evaluation metrics

For test sample \(j\), define the true oracle value:

\[
V^*_{k,j}
=
y_{k,j}^{\top}z^*_{k,j}.
\]

For method \(m\):

\[
V_{m,k,n,s,j}
=
y_{k,j}^{\top}
\widehat z_{m,k,n,s,j}.
\]

### 10.1 Raw decision gap

\[
\operatorname{Gap}_{m,k,n,s,j}
=
V^*_{k,j}
-
V_{m,k,n,s,j}.
\]

### 10.2 Normalized decision gap

\[
\operatorname{NormGap}_{m,k,n,s,j}
=
\frac{
V^*_{k,j}-V_{m,k,n,s,j}
}{
|V^*_{k,j}|+10^{-9}
}.
\]

For presentation, multiply by 100 to obtain percentage points.

### 10.3 Per-run summary

Average over the fixed test set:

```text
mean raw gap
mean normalized gap
median normalized gap
90th and 95th percentile normalized gap
mean achieved/oracle ratio
exact-oracle decision rate
```

### 10.4 Paired method improvement

For each `(topology, train_size, train_seed)`:

\[
\Delta_{k,n,s}
=
\overline{\operatorname{NormGap}}_{\text{2stage},k,n,s}
-
\overline{\operatorname{NormGap}}_{\text{SPO+},k,n,s}.
\]

Report per topology and training size:

```text
mean Delta
median Delta
standard deviation
interquartile range
fraction Delta > 0
fraction Delta above practical threshold
95% confidence interval
```

### 10.5 Across-topology analysis

Topology is the outer unit. Do not treat all test pairs and all train seeds as independent topology replications.

Recommended aggregate analysis:

```text
hierarchical bootstrap:
    sample topologies
    then sample train seeds within each topology

and/or

mixed-effects model:
    Delta ~ log(train_size) + (1 | topology_id)
```

Report both:

```text
per-topology results
aggregate across-topology results
```

## 16. Reproducibility and validity checks

The formal run is invalid if any of the following fails.

### 16.1 Topology invariance

For every sample associated with one topology:

```text
topology_hash is identical
feasible_set_hash is identical
arc_order_hash is identical
number of vertices is identical
number of arcs is identical
```

### 16.2 Seed determinism

Regenerating a sample with the same full key must reproduce:

```text
identical X
identical y
identical sample hash
```

### 16.3 Seed variation

For distinct train seeds, the generated datasets must not all be identical.

In Step2b, explicitly verify:

```text
X differs across seeds
```

Otherwise Step2b has no valid resampling randomness.

### 16.4 Nested training sets

For every topology and train seed:

```text
hash(D_50)  == hash(prefix_50(D_500))
hash(D_100) == hash(prefix_100(D_500))
```

### 16.5 Evaluation reuse

For every topology and regime:

```text
one validation hash
one test hash
```

across all methods, train seeds, and train sizes.

### 16.6 Method pairing

For each paired job:

```text
2stage train_bank_hash == SPO+ train_bank_hash
2stage theta_init == SPO+ theta_init
2stage test_hash == SPO+ test_hash
```

### 16.7 Screening/confirmation separation

No sample key from:

```text
screen_train
screen_validation
screen_test
```

may overlap with:

```text
confirm_train
confirm_validation
confirm_test
```

## 17. Computational considerations

### 17.1 Cache the true oracle

The fixed test set is reused across all trained models. For every `(topology, regime, test_sample)`:

```text
solve true oracle once
cache oracle decision and objective
```

Do not recompute the true oracle for every training seed.

### 17.2 Reuse the fixed feasible structure

For a fixed topology:

```text
cycle/chain candidates are fixed
constraint matrix is fixed
only objective coefficients change
```

Reuse solver models or candidate lists where the implementation permits.

### 17.3 Job granularity

Recommended job unit:

```text
one topology
one train seed
one train size
both methods
```

This gives:

```text
K * 1000 * 3 wrapper jobs
```

For `K=10`:

```text
30,000 wrapper jobs
60,000 method trainings
```

Use:

```text
job manifests
resume support
skip-completed support
per-job logs
atomic output writes
```

### 17.4 Storage

Materializing all 500-sample training banks may be expensive. The preferred formal mode is:

```text
deterministic lazy generation
+
cache on first use
+
store hashes and generator metadata
```

The fixed validation and test sets should be materialized and immutable.

## 22. Final recommended experimental story

```text
Stage 1:
    Screen a broad bank of fixed KEP topologies with a small seed budget.

Stage 2:
    Lock a selected-and-matched set of K fixed topologies.

Stage 3:
    For every topology, generate 1,000 independent nested training banks,
    train 2stage and SPO+ at n = 50, 100, 500, and evaluate every model
    on the same fixed topology-specific validation and test sets.

Stage 4:
    Report full per-topology seed distributions and hierarchical aggregate results.

Stage 5:
    Deeply analyze a small number of confirmed helpful, neutral, and harmful
    graph instances through candidate ranking, oracle landscapes, and critical edges.
```

This design separates:

```text
sample-generation randomness
training-size effects
graph-instance effects
model/solver randomness
```

and provides a defensible path from broad screening to statistically stable DFL case studies.
